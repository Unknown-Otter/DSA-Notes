# 242. Valid Anagram

> [LeetCode 242. Valid Anagram](https://leetcode.com/problems/valid-anagram/)

给定两个字符串 `s` 和 `t`,判断 `t` 是否是 `s` 的字母异位词(anagram)。

**Example 1:**
- Input: `s = "anagram"`, `t = "nagaram"`
- Output: `true`

**Example 2:**
- Input: `s = "rat"`, `t = "car"`
- Output: `false`

---

## 一、解题思路

**核心思路:计数比较**

1. 用一个数据结构记录 `s` 中每个字符出现的次数
2. 遍历 `t`,每遇到一个字符就把对应计数 -1
3. 如果某字符不存在,或计数已经为 0,说明 `t` 多出来字符 → 返回 `false`
4. 遍历完成后,如果计数全为 0,就是 anagram

**前置优化:** 长度不等直接返回 `false`,既能剪枝又能简化后续判断。

---

## 二、解法 1:HashMap

### 代码

```java
class Solution {
    public boolean isAnagram(String s, String t) {
        if (s.length() != t.length()) return false;
        
        Map<Character, Integer> map = new HashMap<>();
        
        // 统计 s 中每个字符
        for (char c : s.toCharArray()) {
            map.put(c, map.getOrDefault(c, 0) + 1);
        }
        
        // 遍历 t,边判断边减
        for (char c : t.toCharArray()) {
            if (!map.containsKey(c) || map.get(c) == 0) {
                return false;
            }
            map.put(c, map.get(c) - 1);
        }
        
        return true;
    }
}
```

### 复杂度

- **时间:** O(n) —— 两次遍历,HashMap 平均 O(1) 操作
- **空间:** O(k) —— k 是不同字符的数量(英文字母场景下其实是 O(1))

> 💡 严格说 HashMap 是**平均** O(1),最坏情况(hash 冲突极端)会退化。Java 8 之后桶里冲突过多会转红黑树,最坏 O(log n)。面试里说 O(n) 总复杂度就够了。

## 三、解法 2:int[26] 数组(更优)

适用于**输入限定为小写英文字母**的场景。

### 版本 A:遍历完再检查(最直观)

```java
class Solution {
    public boolean isAnagram(String s, String t) {
        if (s.length() != t.length()) return false;
        
        int[] count = new int[26];
        
        // 一个循环同时处理:s 中字符 +1,t 中字符 -1
        for (int i = 0; i < s.length(); i++) {
            count[s.charAt(i) - 'a']++;
            count[t.charAt(i) - 'a']--;
        }
        
        // 全 0 才是 anagram
        for (int c : count) {
            if (c != 0) return false;
        }
        
        return true;
    }
}
```

**亮点:** 因为长度相等,可以**一个循环同时处理两个字符串**,代码极简洁。

### 版本 B:边遍历边判断(可提前剪枝)

```java
class Solution {
    public boolean isAnagram(String s, String t) {
        if (s.length() != t.length()) return false;
        
        int[] count = new int[26];
        
        for (char c : s.toCharArray()) {
            count[c - 'a']++;
        }
        
        for (char c : t.toCharArray()) {
            count[c - 'a']--;
            if (count[c - 'a'] < 0) return false;  // t 多了字符
        }
        
        return true;
    }
}
```

> 💡 因为长度相等,如果没有任何字符"多",也不可能有字符"少",所以最后**不需要再检查数组是否全 0**,直接 `return true`。这个推理值得记。

### 复杂度

- **时间:** O(n)
- **空间:** O(1) —— 固定 26 个 int,不随 n 增长

## 四、HashMap vs 数组对比

| | HashMap 版本 | 数组版本 |
|---|---|---|
| 时间 | O(n) 平均 | O(n) 严格 |
| 空间 | O(k) | O(1)(固定 26) |

| 面试好感度 | 一般 | 更优 |

## 五、调试过程中踩的坑(原始代码 → 修复版)

我最初写的代码有 7 个问题,逐个记下来作为避坑提醒:

### 坑 1:Map 初始化少了 `new`
```java
Map<Character, Integer> map = HashMap<>();       // ❌
Map<Character, Integer> map = new HashMap<>();   // ✅
```

### 坑 2:`getOrDefault` 少参数
`getOrDefault(key, defaultValue)` **必须传两个参数**。
```java
map.put(c, map.getOrDefault() + 1);          // ❌
map.put(c, map.getOrDefault(c, 0) + 1);      // ✅
```

### 坑 3:Map 没有 `contains()` 方法 ⚠️
- **`Map`** 用 `containsKey()` / `containsValue()`
- **`Set` / `List`** 才用 `contains()`

```java
if (map.contains(c))      // ❌
if (map.containsKey(c))   // ✅
```

### 坑 4:`map.get(c)--` 不能修改 map ⚠️ 概念问题

`map.get(c)` 返回的是 `Integer`(包装类),是**临时副本**,不是 map 内部存储的引用。而且 `Integer` 是**不可变(immutable)**的,根本不能"原地修改"。

```java
map.get(c)--;                    // ❌ 完全无效
map.put(c, map.get(c) - 1);      // ✅ 取出来,减一,放回去
```

> 💡 这跟 C 指针思维差异很大。在 C 里你拿到的是地址,可以原地改;在 Java 里 `map.get` 返回值的副本,要更新必须显式 `put`。

### 坑 5:if-else 分支顺序错误 ⚠️ 逻辑问题
```java
if (map.contains(c)) {
    map.get(c)--;
} else if (!map.contains(c)) {
    return false;
} else if (map.get(c) == 0) {     // ❌ 永远走不到
    return false;
}
```

前两个 `if-else` 已经覆盖了"包含"和"不包含"两种情况,**第三个分支永远不会被执行**。

应该把"是否包含"和"值是否为 0"**合并判断**:
```java
if (!map.containsKey(c) || map.get(c) == 0) {
    return false;
}
map.put(c, map.get(c) - 1);
```

### 坑 6:函数没有最终 return
方法返回 `boolean`,但循环结束后没有 `return true`,编译都过不了。

### 坑 7:缺少长度判断
如果 `s = "aab"`, `t = "ab"`,只遍历 `t` 是检测不出来的(map 里会剩 `{a: 1}` 但代码直接 return true)。

**解决:** 一开始就 `if (s.length() != t.length()) return false;`

## 六、引申知识点 1:ASCII 与 Unicode

### ASCII
- 1963 年的最早字符编码,**128 个字符**(7 位)
- 只够英文用
- 关键值:`'0' = 48`,`'A' = 65`,`'a' = 97`

### Unicode
- 给世界上每个字符分配一个**唯一编号**(叫 **code point,码点**)
- 范围:`0` ~ `0x10FFFF`(约 **110 万** 个码点)
- 包含所有语言文字 + emoji + 各种符号
- **前 128 个码点和 ASCII 完全一样** → ASCII 是 Unicode 的子集

### Unicode ≠ 编码方式 ⚠️ 关键区分

- **Unicode** = 字符到数字的映射表(`'中' → 20013`)
- **UTF-8 / UTF-16 / UTF-32** = 这些数字怎么存到内存或文件里

打个比方:
- Unicode 像"身份证号",规定每个人有唯一号码
- UTF-x 像"怎么把这个号码写在纸上",有不同格式

| 编码 | 怎么存 | 谁在用 |
|---|---|---|
| **UTF-8** | 变长,1~4 字节 | 网页、Linux、大部分现代系统 |
| **UTF-16** | 变长,2 或 4 字节 | **Java 内部**、Windows |
| **UTF-32** | 固定 4 字节 | 很少用,太占空间 |

## 七、引申知识点 2:Java char 的坑

### Java 的 char 是什么

**Java 的 `char` 是 16 位的,本质是一个 UTF-16 code unit。**

- 16 位能表示 `0 ~ 65535`(`0xFFFF`)
- 这范围叫 **BMP(Basic Multilingual Plane,基本多文种平面)**
- BMP 包含:所有英文、几乎所有常用中文、大部分常用语言文字 ✅

### 问题:BMP 外的字符怎么办?

Unicode 有十几万个字符,远超 65536。BMP 外字符(emoji、罕见汉字、古文字)**一个 char 装不下**。

**Java 的解决方案:用两个 char 表示一个字符,叫代理对(surrogate pair)。**

```java
String s = "😀";
s.length();   // 2 ❗ 不是 1!
```

### 实际后果

```java
String s = "A中😀";
s.length();                          // 4 ❗ (char 数量)
s.codePointCount(0, s.length());     // 3 ✅ (真实字符数)
```

| 字符 | Code point | UTF-16 char | char 数 |
|---|---|---|---|
| `A` | 65 | `0x0041` | 1 |
| `中` | 20013 | `0x4E2D` | 1 |
| `😀` | 128512 | `0xD83D 0xDE00` | **2** |

**所以** `for (char c : s.toCharArray())` 遍历 emoji 字符串时,会把一个 emoji **拆成两半**,每一半都不是合法字符,计数就乱了。

## 八、引申知识点 3:Code Point 详解

### 三大概念区分

| 概念 | 是什么 | Java 类型 | 范围 |
|---|---|---|---|
| **Code point** | Unicode 字符编号 | `int` | 0 ~ 0x10FFFF |
| **Code unit** | UTF-16 存储单位 | `char` | 0 ~ 0xFFFF |
| **Character** | 人类视角的"一个字" | —— | 概念上 |

**关键关系:**
- BMP 内字符:**1 code point = 1 char**
- BMP 外字符:**1 code point = 2 char(代理对)**

### Code Point 常用 API

**1. `String.codePoints()` ⭐ 最推荐**

返回 `IntStream`,**自动正确处理代理对**。
```java
s.codePoints().forEach(cp -> { /* 处理 */ });
s.codePoints().count();       // 真实字符数
s.codePoints().toArray();     // int[] 数组
```

**2. `String.codePointAt(int index)`**

返回指定位置的 code point。注意 `index` 是 char 下标。
```java
int cp = s.codePointAt(0);
```

**3. `String.codePointCount(int begin, int end)`**

统计范围内真实字符数。
```java
s.codePointCount(0, s.length());
```

**4. `Character.toChars(int codePoint)`**

code point 转回 char 数组。
```java
String emoji = new String(Character.toChars(128512));  // "😀"
```

**5. `Character.charCount(int codePoint)`**

告诉你这个 code point 占几个 char(1 或 2)。用于推进遍历下标。
```java
i += Character.charCount(cp);
```

**6. `Character.isXxx(int)` 系列**

`isLetter` / `isDigit` / `isWhitespace` 都有 char 和 int 两个版本。**用 int 版本才能正确处理 BMP 外字符**。

### 三种遍历字符串的姿势

**姿势 1:`codePoints()`(最推荐)** ⭐
```java
s.codePoints().forEach(cp -> {
    // cp 是 int,完整 code point
});
```

**姿势 2:for + codePointAt**
```java
for (int i = 0; i < s.length(); ) {
    int cp = s.codePointAt(i);
    // 处理 cp
    i += Character.charCount(cp);  // BMP 内 +1,BMP 外 +2
}
```

**姿势 3:`for (char c : ...)`(对 emoji ❌)**
```java
for (char c : s.toCharArray()) {
    // emoji 会被拆成两半
}
```

> 💡 LeetCode 大部分题目限定 ASCII 小写字母,这种写法是 OK 的。但面试 follow-up 问到 Unicode 时,要知道它的局限。

## 九、引申知识点 4:ASCII 字符技巧 📌

### 必背数字
```
'0' = 48
'A' = 65   ('A' ~ 'Z' = 65 ~ 90)
'a' = 97   ('a' ~ 'z' = 97 ~ 122)
```

### 常用 idiom
- `c - '0'` → 字符数字转 int(`'5' - '0' = 5`)
- `c - 'a'` → 小写字母转 0~25 下标
- `c - 'A'` → 大写字母转 0~25 下标
- 大写转小写:`c + 32` 或 `c | 32`
- 小写转大写:`c - 32` 或 `c & ~32`

### 为什么 `c - 'a'` 能转下标?

`char` 在 Java 里本质是个整数(Unicode 码点)。`'a' = 97`,`'b' = 98`...`'z' = 122`。两个整数相减,自然得到 `0 ~ 25`。这是个非常常见的 idiom,看到要能立刻反应过来。

## 十一、Follow-up:Unicode 输入怎么办?

### 错误想法 ❌
"用更大的数组,比如 `int[1114112]`"

**问题:**
1. **空间爆炸**:就算输入只有 `"abc"`,也开 100 多万个 int(4MB)
2. **Java char 只有 16 位**,emoji 这种 BMP 外字符用 char 遍历会拆成两半,直接出错

### 正确做法:HashMap + code point

```java
class Solution {
    public boolean isAnagram(String s, String t) {
        if (s.length() != t.length()) return false;
        
        Map<Integer, Integer> map = new HashMap<>();
        
        // 用 codePoints() 正确处理代理对
        s.codePoints().forEach(cp -> 
            map.merge(cp, 1, Integer::sum)
        );
        
        int[] tCps = t.codePoints().toArray();
        for (int cp : tCps) {
            if (!map.containsKey(cp) || map.get(cp) == 0) {
                return false;
            }
            map.put(cp, map.get(cp) - 1);
        }
        
        return true;
    }
}
```

> 💡 `map.merge(key, 1, Integer::sum)` 等价于 `map.put(k, map.getOrDefault(k, 0) + 1)`,但更优雅。**值得记一笔。**

> ⚠️ 注意:这里用 `s.length() != t.length()` 做长度预检查其实不严谨,因为 emoji 字符串的 `length()` 是 char 数量。更严格的做法是比较 `codePointCount`,但对于本题大部分情况下 char 数量相等也够用。

### 面试 Follow-up 模板答案 🎯

> "ASCII 只有 128 个字符,所以可以用 `int[128]` 这样的固定数组。但 Unicode 有十几万个字符,数组开太大不现实。而且 Java 的 char 只有 16 位,只能表示 BMP 内的字符,emoji 这种 BMP 外的字符要用两个 char(代理对)表示,直接 `for (char c : ...)` 遍历会把一个 emoji 拆成两半,计数就错了。所以应该改用 HashMap,key 用 code point(int)而不是 char,通过 `String.codePoints()` 来遍历,这样既能节省空间,也能正确处理所有 Unicode 字符。"

**回答的三个层次:**
1. **识别问题** —— 数组方案为什么不行
2. **知道为什么** —— Java char 16 位 + 代理对
3. **给出方案** —— HashMap + code point + codePoints()

## 十二、最终总结(背诵要点)

1. **基础解法**:计数 → 遍历 → 比较,长度不等先剪枝
2. **数据结构选择**:小字符集用数组(O(1) 空间),大字符集用 HashMap
3. **HashMap 易错点**:`containsKey` 不是 `contains`;`Integer` 不可变,改值必须 `put`;`getOrDefault` 要传两个参数
4. **数组解法的精髓**:`c - 'a'` 转下标;长度相等可以一个循环同时处理两串
5. **ASCII ⊂ Unicode**:Unicode 是超集,前 128 个码点完全相同
6. **Unicode 是字符表,UTF-x 是存储方式**,不要混淆
7. **Java char = UTF-16 code unit**,16 位,只能表示 BMP
8. **emoji 等 BMP 外字符**要用代理对(2 个 char),`length()` 返回的是 char 数,不是字符数
9. **正确处理 Unicode**:用 `codePoints()` 流式 API,以 `int` 作为 key 而不是 `char`
10. **`map.merge(k, 1, Integer::sum)`** 是优雅的计数写法
