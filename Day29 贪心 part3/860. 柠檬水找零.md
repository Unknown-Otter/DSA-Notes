# 860. 柠檬水找零

> [LeetCode 链接](https://leetcode.cn/problems/lemonade-change/)

## 题目简述

每杯柠檬水售价 5 美元，顾客会付 5、10 或 20 美元。一开始手头没有零钱，需要给每位顾客正确找零。能完成所有找零返回 `true`，否则返回 `false`。

---

## 一、我的错误代码与问题分析

### 错误代码

```java
class Solution {
    public boolean lemonadeChange(int[] bills) {
        int income = 0; 
        int change = 0; 

        for (int i = 0; i < bills.length; i++) {
            if (bills[i] == 5) {
                income += bills[i]; 
            } else if (bills[i] > 5) {
                int extra = bills[i] - 5; 
                if (income >= extra) {
                    income += bills[i] - extra; 
                    continue; 
                } else {
                    return false; 
                }
            }
        }
        return true;
    }
}
```

### 错误一：把不同面额的钱混在一起算

我用一个 `income` 变量记录"手上的总金额"，但**找零的时候面额是不能互换的**。

**反例**：`bills = [5, 5, 10, 10, 20]`

按我的逻辑：
1. 收 5 → `income = 5`
2. 收 5 → `income = 10`
3. 收 10，找 5 → `income = 10`（剩一张 5 和一张 10）
4. 收 10，需要找 5，但 `income = 10` 看起来够 → 我的代码会通过，但实际上手里**没有 5 块**了，应该返回 `false`

**根本问题**：20 块不能拆成 5 块用，10 块也不能拆成两个 5 块用。必须**分别记录每种面额的张数**。

### 错误二：金额累加逻辑写歪了

```java
income += bills[i] - extra;  // extra = bills[i] - 5，所以这行等价于 income += 5
```

不管顾客付 10 还是 20，每次都只 `+5`，丢掉了顾客实际付的钱。这一步歪打正着掩盖了一部分问题，但底层逻辑还是错的。

### 错误三：没有体现贪心选择

收到 20 块时，我没区分是用 `10 + 5` 找零还是 `5 + 5 + 5` 找零，但这两种策略对后续找零能力的影响是不一样的。

---

## 二、正确思路（贪心）

可供判断的状态非常少，只需要维护 5 和 10 的张数（20 不用记，因为 20 找不出去）。

分三种情况：

1. **账单是 5**：直接收下，`five++`
2. **账单是 10**：消耗一张 5，增加一张 10
3. **账单是 20**：**优先**消耗 `一张 10 + 一张 5`，不够再消耗 `三张 5`

### 贪心点在情况三

为什么收 20 块时优先用 `10 + 5` 而不是 `5 + 5 + 5`？

> 因为 **10 块只能给 20 找零**，而 **5 块给 10 和 20 都能找零**，5 块更"万能"。

**局部最优**：遇到 20，优先消耗 10，多保留 5。
**全局最优**：完成全部找零。

局部最优可以推出全局最优，且找不到反例，所以贪心成立。

## 三、修正代码

```java
class Solution {
    public boolean lemonadeChange(int[] bills) {
        int five = 0;
        int ten = 0;

        for (int i = 0; i < bills.length; i++) {
            // 情况一：收 5，直接收下
            if (bills[i] == 5) {
                five++;
            } 
            // 情况二：收 10，消耗一张 5
            else if (bills[i] == 10) {
                five--;
                ten++;
            } 
            // 情况三：收 20，优先用 10 + 5，否则用三张 5
            else if (bills[i] == 20) {
                if (ten > 0) {
                    ten--;
                    five--;
                } else {
                    five -= 3;
                }
            }
            // 任意时刻面额数为负，说明找不开
            if (five < 0 || ten < 0) return false;
        }
        return true;
    }
}
```

- **时间复杂度**：O(n)
- **空间复杂度**：O(1)

## 四、总结

1. **核心教训**：涉及不同面额的找零问题，**绝对不能用总金额做判断**，要分别记录每种面额的数量。
2. **分类讨论的力量**：题目看起来复杂，但把所有情况列出来后，会发现可决策的空间其实很小（这里只有三种账单 × 有限的找零方式），逻辑就清晰了。
3. **贪心的判断标准**：当某种资源比另一种资源"更通用 / 更稀缺"时，优先消耗"不通用"的那种，把灵活的留到后面 —— 这是贪心选择的常见模式。
4. **代码小技巧**：在循环内每一步结束后统一判断 `five < 0 || ten < 0`，比在每个分支里单独写 `if 不够 return false` 要简洁。
