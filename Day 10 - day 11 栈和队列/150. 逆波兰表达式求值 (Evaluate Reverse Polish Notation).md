# 150. 逆波兰表达式求值 (Evaluate Reverse Polish Notation)

> [LeetCode 链接](https://leetcode.cn/problems/evaluate-reverse-polish-notation/)

## 核心思路

使用**栈**来处理后缀表达式。栈里存的永远是**"待消费的操作数"** —— 要么是还没被用过的原始数字，要么是已经算出来、等着被下一个操作符使用的**中间结果**。这两者在栈看来**没有区别**。

## 为什么不能用数组索引 (i-2, i-1) 的思路?

一开始很容易想成:"遇到操作符，就取它前面两个位置的数字来算"。但这个思路是**错的**。

反例: `["4","13","5","/","+"]`
- 遇到 `/` 时，前两个确实是 `13` 和 `5` ✓
- 但遇到 `+` 时，前两个是 `5` 和 `/`...❌ `/` 不是数字!

**根本原因**: 操作数和操作符在原数组里**不一定相邻**，中间计算结果会"插入"进来参与后续运算。数组索引表达不了这种关系。

## 栈思维的关键转变

| 数组思维 ❌ | 栈思维 ✅ |
|---|---|
| 关心元素在原数组的**位置** | 关心元素现在是否**待处理** |
| 数字和计算结果是**两种东西** | 数字和计算结果是**同一种东西**(都是"待消费的值") |
| 用 `i-2`, `i-1` 回看 | 用 `pop()` 取最近的待消费值 |

## 识别信号: 什么时候该用栈?

1. **需要"往回看"，但回看距离不固定** —— 写出 `i-2`, `i-1` 时就该警觉
2. **出现"最近的、还没...的 X"** —— 最近的未闭合左括号、最近的两个未使用数字等
3. **嵌套/递归结构** —— 后缀表达式本质是表达式树，栈天然处理子树

## 算法步骤

1. 遍历 `tokens` 数组
2. 如果是**数字**:转成 `int` 后 `push` 进栈
3. 如果是**操作符**:
   - `pop` 出栈顶元素作为**右操作数** `b`
   - 再 `pop` 出栈顶元素作为**左操作数** `a`
   - 计算 `a op b`，将结果 `push` 回栈
4. 遍历结束后，栈里剩下的唯一元素就是答案

## 代码实现

```java
class Solution {
    public int evalRPN(String[] tokens) {
        Deque<Integer> stack = new ArrayDeque<>();

        for (int i = 0; i < tokens.length; i++) {
            String token = tokens[i];

            if (token.equals("+")) {
                int b = stack.pop();
                int a = stack.pop();
                stack.push(a + b);
            } else if (token.equals("-")) {
                int b = stack.pop();
                int a = stack.pop();
                stack.push(a - b);  // 顺序重要: a - b ≠ b - a
            } else if (token.equals("*")) {
                int b = stack.pop();
                int a = stack.pop();
                stack.push(a * b);
            } else if (token.equals("/")) {
                int b = stack.pop();
                int a = stack.pop();
                stack.push(a / b);  // 顺序重要: a / b ≠ b / a
            } else {
                stack.push(Integer.parseInt(token));
            }
        }

        return stack.pop();
    }
}
```

## 关键细节

1. **栈存 `Integer` 不是 `Character`**: token 可能是 `"12"`、`"-3"` 这种多位数/负数，单个 `char` 装不下。用 `Integer.parseInt(token)` 转换。

2. **`String` 比较用 `.equals()` 不能用 `==`**: `tokens[i]` 是 `String` 类型，不能和 `'+'`(char) 直接比较。

3. **pop 顺序**: 栈是 LIFO，**先 pop 出来的是右操作数**，后 pop 出来的是左操作数。
   - 对 `-` 和 `/`:顺序错了结果就错了
   - 对 `+` 和 `*`:满足交换律，但养成统一习惯更好

4. **为什么从 `i=0` 开始**: 不能假设前两个就是数字。例如 `["4","13","5","/","+"]` 需要先把 4、13、5 都入栈。

## 执行过程演示

输入: `["4","13","5","/","+"]`

| 当前 token | 操作 | 栈状态 (栈顶在右) |
|---|---|---|
| `4` | push 4 | [4] |
| `13` | push 13 | [4, 13] |
| `5` | push 5 | [4, 13, 5] |
| `/` | pop 5, pop 13, push 13/5=2 | [4, 2] |
| `+` | pop 2, pop 4, push 4+2=6 | [6] |

返回 `6` ✓

**注意**: 遇到 `+` 时，栈顶的 `2` **根本不在原数组里** —— 它是算出来的中间结果。这就是为什么索引行不通，而栈可以。

## 栈解题通用模板

```
for 每个元素 in 输入:
    if 元素是"数据" (操作数):
        push 进栈              // 先存着，还用不上
    else 元素是"触发器" (操作符/右括号/更大的数...):
        从栈里 pop 出需要的数据
        处理它们
        如果处理结果还要参与后续运算 → push 回栈
```

**同类题目**: 括号匹配、单调栈、字符串解码 `3[a2[c]]`、简化路径等。区别只在于:
- 什么算"数据"
- 什么算"触发器"
- 处理完要不要 push 回去

## 复杂度分析

- **时间复杂度**: `O(n)`，每个 token 处理一次
- **空间复杂度**: `O(n)`，栈最多存 `n/2 + 1` 个操作数

## 一句话总结

> 当"要操作的元素"和"触发操作的元素"在原数组中**距离不固定**、且**中间结果会参与后续运算**时，就不要用索引访问，改用栈 —— 栈里存的是"**待消费的值**"，不区分是原始输入还是计算结果。
