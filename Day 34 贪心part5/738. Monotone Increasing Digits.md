# 738. Monotone Increasing Digits

> [LeetCode 链接](https://leetcode.com/problems/monotone-increasing-digits/)

## 题目理解

一个整数被称为**单调递增的数字**，当且仅当它的每一对相邻数字 `x` 和 `y` 都满足 `x <= y`（从左到右**不下降**）。

给定一个整数 `n`，返回**小于或等于 `n`** 的、数字单调递增的**最大整数**。

**示例：**

| n | 输出 | 说明 |
|---|------|------|
| 10 | 9 | 10 不单调（1 > 0），往下找最大单调数 |
| 1234 | 1234 | 本身就单调，直接返回 |
| 332 | 299 | 332 不单调，最大的单调数是 299 |

**约束：** `0 <= n <= 10^9`（最多 10 位数）

## 思路推导

### 暴力解（会超时）

从 `n` 开始往下递减，逐个检查是否单调，找到第一个就返回。  
**问题：** `n` 最大 `10^9`，最坏情况要试十亿次，**严重超时**。

### 贪心思路：从右往左扫描

**核心观察：** 当发现 `digits[i-1] > digits[i]`（违反单调）时：
1. 把 `digits[i-1]` **减 1**
2. 从位置 `i` 开始，**右边所有位填 9**（让结果尽可能大）

**为什么是减 1 + 填 9？**  
以 `332` 为例：百位的 3 不能保留（否则后两位最多只能是 33，但原数是 32 已经违反了）。所以百位必须降到 2，那后面两位为了让数最大，就填 99 → `299`。

### 关键陷阱：连锁反应

考虑 `333222`：
- 第一个违反点在位置 2 → 3（`3 > 2`）
- 把位置 2 的 `3` 减 1 → 变成 `2`，结果：`3 3 2 9 9 9`
- **但是！** 位置 1 的 `3` 又比位置 2 的 `2` 大了，**前面的单调性被破坏**！
- 所以要继续往前传递：位置 1 也减 1，位置 0 也减 1...
- 最终：**`299999`**

### 为什么从右往左扫能解决连锁反应？

**关键洞察：** 每次把 `digits[i-1]` 减 1 后，**新产生的违反**只可能在**当前位置的左边**（因为我刚减小了当前位，所以它和左邻居的关系才可能出问题）。而扫描方向**正好就是往左走**——所以连锁反应被自然地一路扫过去，**不需要回头**。

## 算法步骤

1. **转换：** 把 int 转成 char 数组（方便逐位修改）
2. **初始化哨兵：** `flag = digits.length`，表示"默认不需要填 9"
3. **从右往左扫描：** `i` 从 `len-1` 到 `1`
   - 如果 `digits[i-1] > digits[i]`：把 `digits[i-1]` 减 1，记录 `flag = i`
4. **统一填 9：** 从 `flag` 位置开始，把右边所有位刷成 `'9'`
5. **转回 int 返回**

## Java 代码

```java
class Solution {
    public int monotoneIncreasingDigits(int n) {
        char[] digits = String.valueOf(n).toCharArray();
        int len = digits.length;
        int flag = len;  // 哨兵值：默认不需要填 9

        // 从右往左扫描，处理违反点
        for (int i = len - 1; i > 0; i--) {
            if (digits[i - 1] > digits[i]) {
                digits[i - 1]--;
                flag = i;
            }
        }

        // 统一把 flag 之后的位填成 '9'
        for (int i = flag; i < len; i++) {
            digits[i] = '9';
        }

        return Integer.parseInt(new String(digits));
    }
}
```

## 关键决策点

- **扫描方向选择：** 从**右往左**而非从左往右，能自然处理连锁回退
- **哨兵值 flag = len：** 用一个不可能的初始值代替显式的"是否发生过违反"的布尔判断
- **减 1 与填 9 分离：** 扫描循环只做"减 1 + 记 flag"，循环结束后再独立填 9，逻辑清晰
- **char 数组操作：** char `'0'` ~ `'9'` 可以直接用 `++` `--` 和 `>` `<` 比较，因为 ASCII 码连续

## 执行过程示例

### 示例 1：`n = 332`

```
初始：    3 3 2     flag = 3
i = 2：   digits[1]=3 > digits[2]=2 ❌
          digits[1]-- → 3 2 2，flag = 2
i = 1：   digits[0]=3 > digits[1]=2 ❌
          digits[0]-- → 2 2 2，flag = 1
扫描结束：2 2 2，flag = 1
填 9：    2 9 9
输出：    299 ✓
```

### 示例 2：`n = 333222`

```
初始：    3 3 3 2 2 2     flag = 6
i = 5：   2 ≤ 2 ✓ 不动
i = 4：   2 ≤ 2 ✓ 不动
i = 3：   3 > 2 ❌  digits[2]-- → 3 3 2 2 2 2，flag = 3
i = 2：   3 > 2 ❌  digits[1]-- → 3 2 2 2 2 2，flag = 2
i = 1：   3 > 2 ❌  digits[0]-- → 2 2 2 2 2 2，flag = 1
扫描结束：2 2 2 2 2 2，flag = 1
填 9：    2 9 9 9 9 9
输出：    299999 ✓
```

### 示例 3：`n = 1234`（本身单调）

```
初始：    1 2 3 4     flag = 4
i = 3：   3 ≤ 4 ✓ 不动
i = 2：   2 ≤ 3 ✓ 不动
i = 1：   1 ≤ 2 ✓ 不动
扫描结束：1 2 3 4，flag = 4（未触发更新）
填 9 循环：i = 4 不满足 i < 4，循环不执行
输出：    1234 ✓
```

## 复杂度分析

- **时间复杂度：** `O(log₁₀ n)`，即数字位数。`n ≤ 10^9` 最多 10 位，实际是 O(10) 常数级
- **空间复杂度：** `O(log₁₀ n)`，char 数组的存储开销

## 类型转换知识点

这道题涉及 `int ↔ String ↔ char[]` 的链式转换，整理如下：

### `String.valueOf(...)` —— 万能"转字符串"
属于 **`String` 类的静态方法**，把任何类型转成字符串。

| 输入类型 | 例子 | 输出 |
|---------|------|------|
| `int` | `String.valueOf(332)` | `"332"` |
| `char[]` | `String.valueOf(new char[]{'a','b'})` | `"ab"` |
| `boolean` | `String.valueOf(true)` | `"true"` |
| `Object` | `String.valueOf(null)` | `"null"`（不报错）|

### `Integer.parseInt(...)` —— 字符串转 int
属于 **`Integer` 类的静态方法**。

```java
Integer.parseInt("332");      // 332
Integer.parseInt("-100");     // -100
Integer.parseInt("09");       // 9（前导 0 自动忽略）
Integer.parseInt("3.14");     // ❌ NumberFormatException
Integer.parseInt("abc");      // ❌ NumberFormatException
Integer.parseInt("FF", 16);   // 255（指定进制）
```

### 转换链路图

```
  int n = 332
       │ String.valueOf(n)
       ▼
  String "332"
       │ .toCharArray()
       ▼
  char[] {'3','3','2'}     ← 在这里逐位修改
       │ new String(digits)
       ▼
  String "299"
       │ Integer.parseInt(...)
       ▼
  int 299
```

### 常用类型转换速查表

| 起点 → 终点 | 写法 |
|------------|------|
| `int` → `String` | `String.valueOf(n)` 或 `Integer.toString(n)` |
| `String` → `int` | `Integer.parseInt(s)` |
| `String` → `char[]` | `s.toCharArray()` |
| `char[]` → `String` | `new String(arr)` 或 `String.valueOf(arr)` |
| `int` → `char` | `(char)(n + '0')` |
| `char` → `int` | `c - '0'` |

## Bug 历史

调试过程中踩过的坑：

1. **循环条件写错：** 写成 `for (int i = len-1; i < len; i--)`，导致 `i` 一直递减为负数无限循环 → 应为 `i > 0`
2. **flag 初始值：** 写成 `flag = len-1`，导致 n 本身单调时最后一位被错误刷成 9 → 应为 `flag = len`（哨兵）
3. **flag 赋值偏移：** 写成 `flag = i-1`，把已经减完的有效位错误地刷成 9 → 应为 `flag = i`
4. **char 与 int 混淆：** 写成 `digits[i] = 9`（整数 9 的 ASCII 是不可见字符）→ 应为 `digits[i] = '9'`
5. **API 拼写错误：** 不存在 `Char.toArray(n)` → 应为 `String.valueOf(n).toCharArray()`

## 总结

1. **识别贪心结构：** 当违反单调时，必须降低高位、抬高低位 → 高位减 1 + 低位填 9
2. **选对扫描方向：** 从右往左扫描能自然处理连锁回退，避免回头扫描
3. **哨兵技巧：** 用 `flag = len` 代替"是否触发过修改"的布尔判断
4. **职责分离：** 扫描循环只做"减 1 + 记 flag"，填 9 在独立循环里完成，逻辑更清晰
5. **char 数组特性：** ASCII 连续，可直接用 `++` `--` 和比较运算符操作字符
