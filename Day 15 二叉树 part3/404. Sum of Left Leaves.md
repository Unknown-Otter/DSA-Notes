# 404. Sum of Left Leaves

> https://leetcode.com/problems/sum-of-left-leaves/

---

## 题目理解

给定一棵二叉树的根节点，返回所有**左叶子**的和。

两个关键词：
- **叶子**：没有左右孩子的节点（`node.left == null && node.right == null`）
- **左**：它是某个节点的左孩子

---

## 递归解法（在父节点处判断）

### 核心思路

不用额外参数，直接在**父节点处**判断：

> "我的左孩子是不是叶子？如果是，就加进去。"

这个思路的关键在于：只有在父节点，你才同时知道两件事：
1. 它是**左孩子**（因为你是从 `root.left` 访问的）
2. 它是不是叶子（检查它自己的 left/right 是否为 null）

### 递归函数的含义

```
sumOfLeftLeaves(root) = "以 root 为根的整棵子树里，所有左叶子的和"
```

这个"整棵子树"拆成三部分：

```
        root
       /    \
    left    right
   subtree  subtree
```

1. **root 这一层自己发现的左叶子** → `sum`（root 的左孩子是叶子）
2. **左子树更深处的左叶子** → `sumOfLeftLeaves(root.left)`
3. **右子树更深处的左叶子** → `sumOfLeftLeaves(root.right)`

三部分没有重叠，加起来就是完整答案。

### 为什么三部分不重叠？

`sum` 只在"左孩子是叶子"时才加，这个叶子节点**进入递归后会直接返回 0**（因为它自己没有左孩子），不会被重复计数。

### 代码

```java
class Solution {
    public int sumOfLeftLeaves(TreeNode root) {
        if (root == null) return 0;

        int sum = 0;
        // 在父节点处判断：左孩子存在 且 左孩子是叶子
        if (root.left != null && root.left.left == null && root.left.right == null) {
            sum += root.left.val;
        }

        int left = sumOfLeftLeaves(root.left);
        int right = sumOfLeftLeaves(root.right);

        return sum + left + right; // ← 三部分加在一起
    }
}
```

---

## 关键问题：为什么必须写 `sum + left + right`？

### 每次调用，`sum` 都是全新的

```java
int sum = 0; // ← 每次进入函数，sum 重新从 0 开始！
```

`sum` **不是全局累加器**，它只活在**当前这一次函数调用**里（局部变量）。

| 调用 | 各自的 sum |
|---|---|
| `f(3)` | `sum = 9` |
| `f(20)` | `sum = 15` |
| `f(9)` | `sum = 0` |

这几个 `sum` 互相完全独立，不会自动合并。

### 递归的信息传递只有一个通道：return 值

```java
return sum + left + right;
//      ↑       ↑      ↑
//    我看到的  左边递归  右边递归
//    左叶子    带回来的  带回来的
```

## 调用栈展开示例

```
        3
       / \
      9   20
         /  \
        15   7
```

| 调用 | sum | left | right | 返回 |
|---|---|---|---|---|
| `f(9)` | 0 | 0 | 0 | **0** |
| `f(15)` | 0 | 0 | 0 | **0** |
| `f(7)` | 0 | 0 | 0 | **0** |
| `f(20)` | 15（15是左叶子）| 0 | 0 | **15** |
| `f(3)` | 9（9是左叶子）| 0 | 15 | **24** ✓ |

每一层只负责"**我能直接看到的左叶子**"，深处的交给递归，最后全部加起来。

### 多层嵌套时的调用栈

```
        3
       / \
      9   20
         /  \
        15   7
       /
      6       ← 6 才是左叶子，不是 15
```

```
f(3) 开始
  sum = 9，等待...
  │
  ├── f(9) → 返回 0
  │
  └── f(20) 开始
        sum = 0（15不是叶子！），等待...
        │
        ├── f(15) 开始
        │     sum = 6（6是左叶子）
        │     ├── f(6)  → 返回 0
        │     └── f(null) → 返回 0
        │     return 6 + 0 + 0 = 6
        │
        └── f(7) → 返回 0

        return 0 + 6 + 0 = 6

  return 9 + 0 + 6 = 15 ✓
```

**每一层的 `sum` 存在自己的栈帧里，一直等着下面的递归结束，结果通过 `return` 往上传。越深的越先算完，结果一层一层往上交。**

## 迭代解法（显式栈模拟 DFS）

### 思路

用栈手动模拟递归过程，遍历时同样在**父节点处**判断左孩子是否是叶子。

```java
class Solution {
    public int sumOfLeftLeaves(TreeNode root) {
        if (root == null) return 0;

        Stack<TreeNode> stack = new Stack<>();
        stack.add(root);
        int res = 0;

        while (!stack.isEmpty()) {
            TreeNode node = stack.pop();

            // 在父节点处判断：左孩子存在 且 左孩子是叶子
            if (node.left != null && node.left.left == null && node.left.right == null) {
                res += node.left.val;
            }

            // 先压右再压左，保证左边先处理
            if (node.right != null) stack.add(node.right);
            if (node.left != null) stack.add(node.left);
        }

        return res;
    }
}
```

### 为什么先压右再压左？

栈是 LIFO（后进先出），先压右后压左，左边会先被弹出处理，保持和递归一致的遍历顺序。

---

## 两种解法对比

| | 递归 | 迭代 |
|---|---|---|
| 判断逻辑 | 父节点处判断 | 父节点处判断（完全一致）|
| 状态保存 | 隐式调用栈（JVM 管理）| 显式 Stack |
| 代码量 | 更简洁 | 稍多 |
| 栈溢出风险 | 树极深时有风险 | 无风险 |

---

## 总结

1. **判断左叶子的时机**：在**父节点处**判断，同时具备"左孩子"和"是叶子"两个信息
2. **递归局部变量**：每次调用都有独立的 `sum`，信息只能通过 `return` 传递
3. **`return sum + left + right`**：三部分分别对应当前层、左子树深处、右子树深处的左叶子，无重叠
4. **调用栈**：深的先算完，结果一层一层往上交，上面的栈帧一直在等
