# 376. Wiggle Subsequence

> https://leetcode.com/problems/wiggle-subsequence/

## 题意理解

摆动序列：相邻元素之差**严格交替**正负。求最长摆动子序列的长度。

```
[1, 7, 4, 9, 2, 5]  差值为 (6, -3, 5, -7, 3) → 交替正负 → 长度6 ✅
[1, 4, 7, 2, 5]     差值为 (3, 3, -5, 3)      → 前两个都正 ❌
[1, 7, 4, 5, 5]     差值为 (6, -3, 1, 0)       → 最后为零 ❌
```

---

## 贪心思路：数峰谷个数

**核心洞察**：不用真的删元素，只需要数有多少个**局部峰值（peak）和谷值（valley）**。

```
    17        15
   /    \    /  \      16
  /      \  /    \    /  \
1         5      10  /    8
                   \/
                    5
```

每一个转折点（峰或谷）都是贡献，一个都不能跳过。

---

## 两个核心变量

```
curDiff = nums[i+1] - nums[i]   ← 当前这一步的方向
preDiff = 上一次触发摆动时的curDiff  ← 历史方向记录
```

**关键**：`preDiff` 不是 `nums[i] - nums[i-1]`，而是**上一次发生摆动时**的 curDiff。
平坡时 curDiff=0，不触发更新，preDiff **冻结**，保留上一次真实方向。

---

## 所有符号组合穷举

| preDiff | curDiff | 含义 | 触发？ |
|---------|---------|------|--------|
| + | + | 连续上升 | ❌ |
| + | 0 | 上升后进入平坡 | ❌ |
| **+** | **-** | **峰** | **✅** |
| **-** | **+** | **谷** | **✅** |
| - | 0 | 下降后进入平坡 | ❌ |
| - | - | 连续下降 | ❌ |
| **0** | **+** | **初始 / 平坡后上升** | **✅** |
| 0 | 0 | 全平 | ❌ |
| **0** | **-** | **初始 / 平坡后下降** | **✅** |

触发的行：`(+,-)` `(-,+)` `(0,+)` `(0,-)`

合并成条件：
```
(curDiff > 0 && preDiff <= 0)   ← 谷（含平坡末端 + 首尾）
(curDiff < 0 && preDiff >= 0)   ← 峰（含平坡末端 + 首尾）
```

> **设计原则**：不要想"哪些情况要避免"，而是"哪些情况应该触发"，穷举后合并。

---

## 三种平坡情况详解

### 情况一：峰顶有平台

```
[1, 3, 3, 3, 1]
平坡中间：preDiff>0, curDiff=0  → 不触发 ✅
平坡末端：preDiff>0, curDiff<0  → 触发（preDiff>=0 覆盖此情况）✅
```

### 情况二：谷底有平台

```
[3, 1, 1, 1, 3]
平坡中间：preDiff<0, curDiff=0  → 不触发 ✅
平坡末端：preDiff<0, curDiff>0  → 触发（preDiff<=0 覆盖此情况）✅
```

### 情况三：单调坡中有平台（最容易误判！）

```
[1, 2, 2, 2, 3]
上升段：  preDiff=0,  curDiff>0  → 触发，preDiff更新为正数 ✅
平坡中间：preDiff>0,  curDiff=0  → 不触发，preDiff冻结 ✅
平坡末端：preDiff>0,  curDiff>0  → 不触发 ✅
```

> **为什么不误判**：preDiff 记住了进入平坡前的方向（正数），平坡末端 `preDiff>0 && curDiff>0` 不满足任何触发条件。

**情况二 vs 情况三 的本质区别**：

```
谷底平台末端：  preDiff < 0（进入平台前是下降的）→ 触发
单调上升平台末端：preDiff > 0（进入平台前是上升的）→ 不触发
```

两者 curDiff 都是正数，但 preDiff 的历史记录不同，自然区分。

### 情况四：首尾特殊情况

```
[2, 5]  只有两个元素
preDiff 初始 = 0，模拟"第一个数左边有一个与它相等的数"
→ preDiff=0, curDiff>0 → 满足 preDiff<=0 → 触发 ✅
结果 = 2 ✅
```

---

## 接力棒模型

```
第一次循环：
  curDiff = 算出来的值
  触发 → count++，preDiff = curDiff   ← curDiff 交棒给 preDiff

第二次循环：
  curDiff = 新算出来的值
  用新 curDiff 和上一次的 preDiff 比较
  触发 → count++，preDiff = curDiff   ← 再次交棒

以此类推...
```

`preDiff` 永远记录上一次触发时的方向，`curDiff` 永远是当前这步的方向，每次触发后 curDiff 交棒给 preDiff。

---

## 最终代码

```java
class Solution {
    public int wiggleMaxLength(int[] nums) {
        int preDiff = 0;   // 初始为0，模拟第一个数左边有相等元素
        int curDiff = 0;
        int count = 1;     // 默认右端点算一个峰

        for (int i = 0; i < nums.length - 1; i++) {
            curDiff = nums[i + 1] - nums[i];

            if (curDiff > 0 && preDiff <= 0) {   // 谷
                count++;
                preDiff = curDiff;   // 只在触发时更新！
            } else if (curDiff < 0 && preDiff >= 0) {   // 峰
                count++;
                preDiff = curDiff;
            }
            // curDiff == 0 时什么都不做，preDiff 冻结
        }
        return count;
    }
}
```

**时间复杂度**：O(n)
**空间复杂度**：O(1)

## 与 up/down DP 解法对比

另一种等价写法：

```java
class Solution {
    public int wiggleMaxLength(int[] nums) {
        int up = 1, down = 1;
        for (int i = 1; i < nums.length; i++) {
            if (nums[i] > nums[i - 1]) up = down + 1;
            else if (nums[i] < nums[i - 1]) down = up + 1;
            // 相等时什么都不做
        }
        return Math.max(up, down);
    }
}
```

| | preDiff/curDiff 贪心 | up/down |
|---|---|---|
| 思维方式 | 数峰谷个数 | 对偶状态DP |
| 平坡处理 | preDiff 冻结 | 什么都不做 |
| 本质 | 相同 | 相同 |

> up/down 解法是空间优化后的 DP，preDiff/curDiff 更贴近"数峰谷"的直觉。**很多贪心解法本质上是 DP 的极度简化。**
