# 53. Maximum Subarray

> [LeetCode 53. Maximum Subarray](https://leetcode.com/problems/maximum-subarray/)

给定一个整数数组 `nums`，找出和最大的连续子数组，返回其和。

```
Input: nums = [-2,1,-3,4,-1,2,1,-5,4]
Output: 6
Explanation: [4,-1,2,1] 的和最大，为 6
```

---

## 一、最初的思路：滑动窗口？

### 我的初始直觉

> 应该建立一个动态的滑动窗口，从第一个正数开始，向右扩展。一旦当前 sum 变成负数，就把起点向右移动一位。

### 为什么滑动窗口不太合适

**滑动窗口的核心前提**：窗口的扩张/收缩有单调性。

- 比如"和 ≤ target 的最长子数组"问题——元素全是正数时，扩张窗口和单调增，收缩单调减，所以可以双指针
- 但本题数组里有**负数**，扩张窗口时和可能变大也可能变小，收缩时同理
- 左右指针没有清晰的移动规则，套滑动窗口模板会很别扭

### 初始思路里对的部分

- ✓ "sum 变负就抛弃前面" 这个直觉**完全正确**
- ✗ "从第一个正数开始"不对（全负数数组没有正数起点）
- ✗ "起点右移一位" 不准确（应该是直接跳到当前位置重启）

---

## 二、解法一：动态规划（Kadane 算法）

### 状态定义

> `dp[i]` = **以 `nums[i]` 结尾**的最大子数组和

**关键**："以 i 结尾"这个限定强制子数组必须包含 `nums[i]`，让状态转移变得干净。

### 状态转移方程

对于 `nums[i]`，要么接在前面 `dp[i-1]` 后面，要么自己单独成一段：

```java
dp[i] = Math.max(dp[i-1] + nums[i], nums[i]);
```

化简：`dp[i] = Math.max(dp[i-1], 0) + nums[i]`——如果前面累积是负的，就丢掉（视作 0），从自己重新开始。

### 最终答案

`max(dp[0], dp[1], ..., dp[n-1])`，**不是 `dp[n-1]`**，因为最大子数组不一定以最后一个元素结尾。

### 空间优化版代码

```java
class Solution {
    public int maxSubArray(int[] nums) {
        int curSum = nums[0];   // dp[i]：以 nums[i] 结尾的最大和
        int maxSum = nums[0];   // 全局最大值
        
        for (int i = 1; i < nums.length; i++) {
            curSum = Math.max(curSum + nums[i], nums[i]);
            maxSum = Math.max(maxSum, curSum);
        }
        
        return maxSum;
    }
}
```

## 三、解法二：贪心算法

### 贪心策略

维护当前累积和 `curSum`，从左到右遍历：

- `curSum > 0`：前面这段对后面是**正贡献**，留着，继续累加 `nums[i]`
- `curSum <= 0`：前面这段是**负贡献或零贡献**，**直接丢弃**，从 `nums[i]` 重新开始

每一步都更新全局最大值 `maxSum`。

### Java 代码

```java
class Solution {
    public int maxSubArray(int[] nums) {
        int currSum = 0;
        int max = Integer.MIN_VALUE;  // 处理全负数情况
        
        for (int num : nums) {
            currSum += num;                    // 1. 加
            max = Math.max(currSum, max);      // 2. 比
            if (currSum < 0) currSum = 0;      // 3. 断
        }
        
        return max;
    }
}
```

### 三步口诀：**加 → 比 → 断**

1. **加**：`currSum += nums[i]` —— "如果带着前面这段走到下一个位置，累积是多少？"
2. **比**：更新 `max` —— "破纪录了吗？"
3. **断**：若 `currSum < 0` 则清零 —— "这段还值得继续吗？"

### 顺序很关键！

**先加 → 再比 → 最后断**。如果先清零再更新 `max`，会漏掉"以当前元素结尾"的最大值，全负数数组会出错。

## 四、关键概念辨析

### 4.1 为什么是"前面对后面的贡献"，而不是"后面抵消前面"？

#### 核心区别：算法能"决定"什么

遍历到位置 `i` 时，手里有两样东西：
- `currSum`：前面累积（**已知**）
- `nums[i]`：当前元素（**已知**）
- `nums[i+1], ...`：后面的元素（在贪心视角下当作**未知**）

贪心做的是**局部决策**：能问的问题只有"我手里这个 `currSum`，要不要带着继续往后走？"

#### "后面来个 -30 抵消前面" 的疑问

例子：`[5, 3, 2, -30, ...]`

走到 `-30` 时，`currSum = 10`，加上 -30 后 `currSum = -20`。**但在加 -30 之前，`max` 已经被更新成 10 了！**

```java
currSum += num;                      // currSum = -20
max = Math.max(currSum, max);        // max 仍然是 10
if (currSum < 0) currSum = 0;        // 抛弃这段
```

**关键洞察**：

| 变量 | 角色 |
|------|------|
| `currSum` | "还在尝试的当前段"——会被负数拖垮、被清零 |
| `max` | "已经锁定的历史最佳"——只增不减，永远不会被后续负数夺走 |

后面的负数只能影响 `currSum`，**影响不到 `max`**。

#### 形式化解释"前面对后面的贡献"

考虑两种走法，都以 `nums[i+1]` 开始一段新尝试：

- **走法 A**：带着前面的 `currSum`，子数组 = `[起点, ..., i, i+1, ...]`
- **走法 B**：抛弃前面，从 `i+1` 单独开始：`[i+1, ...]`

无论后面发生什么，**A 和 B 后面经历的部分完全一样**，差距永远是 `currSum`。

- `currSum > 0`：A 永远比 B 多赚 `currSum`，**带着走更优**
- `currSum <= 0`：A 永远比 B 差或相等，**抛弃更优**

### 4.2 一次遍历就够了，不需要循环走

走到数组末尾就结束，O(n) 一遍过。

**为什么一次就够？** 遍历过程中，**所有"以某个位置结尾"的最大子数组和都被考虑过**：
- 对每个位置 `i`，那一刻的 `currSum` 就是"以 `nums[i]` 结尾的最大子数组和"
- 最大子数组**一定**以某个位置结尾
- 所以遍历过程中的每个 `currSum` 都被 `max` 比较过 → **不会漏**

> **n 个候选答案，全都参与了比较。**

### 4.3 为什么局部最优能推出全局最优？

#### 贪心成立的两个性质

**性质一：贪心选择性质（Greedy Choice Property）**

> 每一步的局部决策，永远不会让全局答案变差。

**反证法**：假设全局最优子数组是 `nums[L..R]`，和为 `S*`。证明贪心一定经过它：

考虑 `currSum` 在 `L..R` 中间某个位置 `k` 是否会变负被清零？

- 如果 `nums[L..k]` 的和 < 0，那 `nums[k+1..R]` 的和 = `S* - nums[L..k] > S*`
- 这意味着 `nums[k+1..R]` 比 `nums[L..R]` 还大——**与最优解假设矛盾**

**结论**：最优子数组**内部不会触发清零**，`currSum` 一路累加到 `S*`，被 `max` 捕获。

**性质二：最优子结构（Optimal Substructure）**

> "整个数组的最大子数组和" = max("以位置 i 结尾的最大子数组和" for i in 0..n-1)

每个小问题只依赖前一个：`f(i) = max(f(i-1) + nums[i], nums[i])`

#### 一句话总结

> 局部最优能推出全局最优，**当且仅当**局部决策永远不会"误杀"任何潜在的全局最优解。在 53 题里，"清零"操作只丢弃负累积，而**真正的最优子数组内部不可能存在负前缀和**——所以清零绝不会误伤最优解。

### 4.4 贪心 vs DP：算法范式辨析

#### 共同前提

| 性质 | 含义 |
|------|------|
| **最优子结构** | 大问题最优解可由小问题最优解组合得到 |
| **子问题独立性** | 子问题之间互不干扰 |

⚠️ **这两条是必要条件，不是充分条件**——满足这两条只说明问题"有结构"，但用什么算法还看更细的性质。

#### 三种算法的真正分界

| 算法 | 子问题关系 | 决策方式 | 例子 |
|------|----------|---------|------|
| **分治** | 完全独立、不重叠 | 递归解决，合并结果 | 归并排序 |
| **DP** | 有重叠，需要记忆 | 枚举所有转移，取最优 | 53 题 DP 解、背包 |
| **贪心** | 每步只产生一个子问题 | 局部最优决策，**不回头、不枚举** | 53 题贪心解 |

#### 真正区分贪心的核心：**贪心选择性质**

> 每一步的局部最优决策，**不需要枚举其他选项**，就能得到全局最优。

#### 用 53 题对比两种解法

**DP 视角（枚举两个选项）**：
```
f(i) = max( f(i-1) + nums[i],   // 选项 A：接着前面走
            nums[i] )            // 选项 B：自己重新开始
```

**贪心视角（只做一个决策）**：
```java
if (currSum < 0) currSum = 0;   // 直接抛弃，不考虑"万一带着更好"
currSum += nums[i];
```

**贪心比 DP 更"勇"**——它跳过了枚举，直接做决定。这种"勇"只有在能证明**当前一步的最优决策一定是全局最优的一部分**时才合法。

#### 反例：什么时候贪心会错？

**0/1 背包问题**：物品有重量和价值，背包有容量限制，求最大价值。

| 物品 | 重量 | 价值 |
|------|------|------|
| A | 1 | 1 |
| B | 50 | 50 |

容量 50，性价比都是 1。按性价比贪心可能先选 A，剩 49 容量装不下 B → **总价值 1**。
最优解：直接选 B → **50**。**贪心错了**。

问题出在：0/1 背包不满足贪心选择性质——"当前选最高性价比"会切断后面更好的可能性。

#### 判断"这题能不能贪心"的流程

1. 验证最优子结构（大问题能切小问题？）
2. 尝试构造贪心策略
3. **关键：尝试找反例**
   - 找不到反例 → 尝试证明贪心选择性质 → 写代码
   - 找到反例 → 退回 DP

## 五、踩坑记录：错误代码 → 正确代码

### 5.1 错误版本

```java
class Solution {
    public int maxSubArray(int[] nums) {
        int currSum = 0;
        
        for (int i = 0; i < nums.length; i++) {
            currSum += nums[i]; 
            
            if (currSum < 0) {
                currSum = nums[i+1];   // ❌ 错误
                continue;
            }
        }
        
        return currSum;   // ❌ 错误
    }
}
```

### 5.2 错误分析

#### 错误一：没有维护 `max`，返回的不是答案

`return currSum` 返回的是**当前累积和**，不是**历史最大值**。

反例：`[5, -10, 3]`
- i=0：currSum = 5（这才是答案！）
- i=1：currSum = -5，触发清零
- i=2：currSum = 3
- 返回 3，但正确答案是 **5**

**最大子数组和可能在中途出现，然后被后面的负数拖下来**，必须用 `max` 单独记录历史最佳。

#### 错误二：`currSum = nums[i+1]` 越界 + 重复加

```java
if (currSum < 0) {
    currSum = nums[i+1];   // ❌ 越界 + 逻辑错
    continue;
}
```

- **越界**：`i = nums.length - 1` 时 `nums[i+1]` 抛 `ArrayIndexOutOfBoundsException`
- **重复加**：`continue` 进入下一轮后，`currSum += nums[i+1]` 又会执行一次，`nums[i+1]` 被加了两次

正确做法：**`currSum = 0`**，让下一轮循环自然把下一个元素加进来。

#### 错误三：全负数数组得到 0

`[-3, -2, -1]`：每轮 currSum 加完都 < 0 被清零，最终返回 0。
但题目要求子数组**非空**，正确答案应该是 -1。

修复：`max` 初始化为 `Integer.MIN_VALUE`。

#### 错误四：清零顺序问题

清零必须在 `max` 更新之后。如果先清零再更新 max，全负数数组永远更新不到 max。

### 5.3 修正后的正确版本

```java
class Solution {
    public int maxSubArray(int[] nums) {
        int currSum = 0;
        int max = Integer.MIN_VALUE;
        
        for (int i = 0; i < nums.length; i++) {
            currSum += nums[i]; 
            max = Math.max(currSum, max);
            
            if (currSum < 0) {
                currSum = 0;
                continue;   // 这行其实可以删掉（空操作）
            }
        }
        
        return max; 
    }
}
```

### 5.4 进一步精简（面试推荐版本）

```java
class Solution {
    public int maxSubArray(int[] nums) {
        int currSum = 0;
        int max = Integer.MIN_VALUE;
        
        for (int num : nums) {
            currSum += num;
            max = Math.max(currSum, max);
            if (currSum < 0) currSum = 0;
        }
        
        return max;
    }
}
```

改进：
- 删除多余的 `continue`（空操作）
- 用增强 for 循环（不需要下标 `i`）

## 七、面试回答模板

如果面试被考到这道题，可以按这个结构组织回答：

1. **识别问题**：求最大连续子数组和。暴力 O(n²)，但有 O(n) 最优解
2. **算法选择**：Kadane 算法（贪心 / 一维 DP）
3. **核心思路**："维护当前累积和 `currSum` 和历史最大 `max`。`currSum` 变负就抛弃前面，从下一个元素重新开始；`max` 记录全程最高点。"
4. **正确性证明**（如被追问）："最优子数组内部的任何前缀和都必须 ≥ 0，否则可以去掉这个负前缀得到更大的子数组——所以'清零'操作不会切断任何最优解。"
5. **复杂度**：O(n) 时间，O(1) 空间
6. **边界情况**：全负数 → `max` 初始化为 `Integer.MIN_VALUE` 保证非空

## 八、最终总结：一道题的完整思考链条

1. **直觉阶段**：滑动窗口？方向不太对，但抓到了"起点要移动"的关键
2. **DP 视角**：`dp[i] = max(dp[i-1] + nums[i], nums[i])`，"以 i 结尾"是关键
3. **贪心视角**：`currSum < 0` 就抛弃，加→比→断三步走
4. **质疑视角方向**：为什么是"前→后贡献"而不是"后→前抵消"？因为算法做的是基于已知信息的局部决策
5. **确认遍历方式**：一次遍历就够，不需要循环走
6. **追问正确性**：贪心选择性质 + 最优子结构 = 局部最优推全局最优
7. **算法范式辨析**：贪心 ≠ DP，区别在贪心选择性质（不需枚举）
8. **代码实现**：第一版有 4 个错误，修正后正确

### 关键 takeaways

- **`currSum` 是当前尝试，`max` 是历史最佳**，必须分开维护
- **加 → 比 → 断**，顺序不能错
- **贪心成立的关键不是"小问题独立"，而是"贪心选择性质"**——局部决策不切断全局最优
- **判断能否贪心：尝试找反例**。找不到再尝试证明
