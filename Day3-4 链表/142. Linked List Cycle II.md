# 142. Linked List Cycle II

> [LeetCode 142 - Linked List Cycle II](https://leetcode.com/problems/linked-list-cycle-ii/)

给定一个链表的头结点 `head`,返回链表开始入环的第一个节点。如果链表无环,则返回 `null`。**不允许修改链表**。

---

## 方法一:HashSet 哈希表法

### 思路

最直观的想法:用一个 `HashSet` 记录所有访问过的节点。遍历链表时,如果某个节点已经在 set 中,说明这就是入环的第一个节点;如果遍历到 `null`,说明无环。

### 代码

```java
public class Solution {
    public ListNode detectCycle(ListNode head) {
        Set<ListNode> hashset = new HashSet<>();
        ListNode curr = head;

        while (curr != null) {
            if (hashset.contains(curr)) {
                return curr;
            }
            hashset.add(curr);
            curr = curr.next;
        }

        return null;
    }
}
```

### ⚠️ 踩坑记录:`while` 条件该写 `curr != null` 还是 `curr.next != null`?

**最初的错误写法**:

```java
while (curr.next != null) {  // ❌ 报错:NullPointerException
    ...
}
```

**报错信息**:

```
java.lang.NullPointerException: Cannot read field "next" because "<local3>" is null
```

**错误的归因**:一开始以为是"测试用例大部分有环,所以会卡在环里导致报错"。这个想法是**错的**,原因有两点:

1. **环不会导致 NPE,只会导致 TLE(死循环)**。NPE 和 TLE 是两种完全不同的错误。
2. **HashSet 已经保护了循环不会卡在环里**:第二次遇到同一个节点时,`hashset.contains(curr)` 为 true,直接 `return` 退出。

**真正的原因**:问题不在 `.next`,而在 **`curr` 本身可能是 null**。

当测试用例是空链表 `head = []` 时:

```java
ListNode curr = head;        // curr = null
while (curr.next != null)    // 💥 null.next → NPE
```

**关键区分**:`curr.next` 作为表达式被读取时,需要 `curr` 不是 null(否则没有对象可供读取字段)。这是隐含要求,但 `while` 条件本身没有保护它。

### ✅ 正确模板:Java 链表遍历的黄金条件

| 操作 | 安全前提 |
|---|---|
| 读取 `curr.next`(作为右值) | `curr != null` |
| 判断 `curr.next != null` | `curr != null`(隐含!) |
| 赋值 `curr = curr.next` | `curr != null` |

**结论**:`while (curr != null)` 自带保护——它先确认 `curr` 存在,再让循环体安全访问 `curr.next`、`curr.val`、`curr = curr.next` 等所有 `curr.xxx` 操作。

```java
while (curr != null) {
    // 安全访问 curr 的任何字段
    curr = curr.next;  // 即使 curr.next 是 null 也没关系,下一轮 while 会退出
}
```

### 复杂度分析

- **时间复杂度**:`O(n)`,每个节点最多被访问两次(一次 add,一次 contains)。
- **空间复杂度**:`O(n)`,HashSet 最坏情况下存储所有节点。

## 方法二:Floyd 快慢指针(龟兔赛跑算法)

### Intuition:为什么快慢指针一定会相遇?

**第一步:为什么需要"环"才能相遇?**

如果链表是一条直线,fast 走得快,slow 走得慢,fast 会一路领先直到走出链表,**永远不可能相遇**。

只有当 fast 进入环开始转圈,slow 也进入环之后,才有可能相遇。

**第二步:为什么相遇时一定不会"跳过去"?**

设 fast 每次走 2 步,slow 每次走 1 步。当 slow 进入环后,fast 在环里追 slow,此时:

> **fast 相对于 slow 的速度 = 2 - 1 = 1**

也就是说,**如果把 slow 看作静止,fast 是每次一步一步靠近 slow** —— 一格一格挪动,绝对不会跳过 slow。所以两者必定相遇。✨(这一步还有点浪漫:fast 永远会以最温柔的速度追上 slow)

### Intuition:为什么相遇后再走能找到环入口?

设:

- `x` = 头结点到环入口的距离
- `y` = 环入口到相遇点的距离
- `z` = 相遇点回到环入口的距离(沿环方向)

```
        x              y
head ──────────► entry ──────► meet
                  ▲              │
                  │      z       │
                  └──────────────┘
```

**相遇时**:

- slow 走过的节点数:`x + y`
- fast 走过的节点数:`x + y + n(y + z)`,其中 `n` 是 fast 在环里多转的圈数,`(y + z)` 是环的周长

因为 fast 速度是 slow 的 2 倍,所以 fast 走的路程 = slow 的 2 倍:

```
2(x + y) = x + y + n(y + z)
```

两边消去 `(x + y)`:

```
x + y = n(y + z)
```

求 `x`(头到环入口的距离),把 `x` 单独放左边:

```
x = n(y + z) - y
  = (n - 1)(y + z) + z
```

注意 `n ≥ 1`(fast 至少转一圈才能追上 slow)。

**当 n = 1 时**:

```
x = z
```

这意味着:**从 head 走 x 步到环入口** = **从相遇点走 z 步到环入口**。

**当 n > 1 时**:从相遇点出发的指针会在环里多转 `(n-1)` 圈,但**最终的落脚点仍然是环入口**,因为多转的圈不影响终点。

### 算法步骤

1. 用快慢指针找相遇点(若 fast 走到 null,说明无环)。
2. 一个指针从 `head` 出发,另一个从相遇点出发,**两者都每次走一步**。
3. 它们相遇的位置就是环的入口。

### 代码

```java
public class Solution {
    public ListNode detectCycle(ListNode head) {
        ListNode slow = head;
        ListNode fast = head;

        while (fast != null && fast.next != null) {
            slow = slow.next;
            fast = fast.next.next;

            if (slow == fast) {  // 有环,开始找入口
                ListNode index1 = fast;
                ListNode index2 = head;
                while (index1 != index2) {
                    index1 = index1.next;
                    index2 = index2.next;
                }
                return index1;  // 相遇点即为环入口
            }
        }

        return null;  // fast 走到 null,无环
    }
}
```

### 为什么 while 条件是 `fast != null && fast.next != null`?

因为 `fast = fast.next.next` 这一步要连续访问两次 `next`,必须保证:

- `fast != null` —— 否则 `fast.next` 报 NPE
- `fast.next != null` —— 否则 `fast.next.next` 报 NPE

这正是上面讲的"循环体内部需要访问什么,就在 while 条件里保护什么"原则的应用。slow 不需要单独检查,因为 fast 跑得更快,fast 没事 slow 一定没事。

### 复杂度分析

- **时间复杂度**:`O(n)`
  - 第一阶段(找相遇点):slow 最多走 n 步就会被 fast 追上,所以是 `O(n)`。
  - 第二阶段(找入口):最多走 `x` 步,`x ≤ n`,所以也是 `O(n)`。
- **空间复杂度**:`O(1)`,只用了两个指针,与链表长度无关。

## 两种方法对比

| 维度 | HashSet 法 | Floyd 快慢指针 |
|---|---|---|
| 时间复杂度 | `O(n)` | `O(n)` |
| 空间复杂度 | `O(n)` | `O(1)` ✅ |
| 思路难度 | 简单直观 | 需要数学推导 |
| 代码量 | 少 | 中等 |
| 面试推荐 | 作为 brute force 先讲 | **最优解,必须掌握** |

**面试策略**:先说 HashSet 解法作为 baseline,讲清楚 `O(n)` 空间;然后主动提出可以用 Floyd 算法优化到 `O(1)` 空间,展示对最优解的掌握。

---

## 总结(Key Takeaways)

1. **Java 链表遍历模板**:默认用 `while (curr != null)`,只有当循环体内部需要访问 `curr.next.xxx` 时才加 `&& curr.next != null`。
2. **NPE 的根因永远是"对 null 调用方法/字段"**,debug 时要找的是"哪个变量在那一刻是 null",而不是凭直觉归因到题目特征(如"环")。
3. **Floyd 算法两个关键直觉**:
   - 相对速度为 1,fast 一步一步追上 slow,不会跳过去。
   - 数学推导出 `x = (n-1)(y+z) + z`,所以从 head 和相遇点同速出发,会在环入口相遇。
4. **环不会导致 NPE,只会导致 TLE**。区分这两种错误是 debug 的基本功。
