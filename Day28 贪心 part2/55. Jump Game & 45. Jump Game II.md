# 55. Jump Game

## 题目描述

给定一个非负整数数组 `nums`,初始位于数组的第一个下标。数组中的每个元素代表你在该位置可以跳跃的**最大长度**。

**返回**:`true` 如果可以到达最后一个下标,否则返回 `false`。

**示例**:
```
Input:  nums = [2,3,1,1,4]
Output: true   // 0 -> 1 -> 4

Input:  nums = [3,2,1,0,4]
Output: false  // 卡在 index 3,nums[3]=0 跳不动
```

## 我的思路:逆向贪心(从后往前推)

### 核心观察

**算每个位置到最后一位的距离**,如果 `nums[i] >= 距离`,说明从位置 `i` 出发可以**直接跳到(或越过)终点**。

举例 `nums = [2,3,1,1,4]`:
```
索引:    0  1  2  3  4
nums:   [2, 3, 1, 1, 4]
距离:   [4, 3, 2, 1, 0]
```

那么问题就转化为:**能不能从开头跳到这个位置 `i`?**

### 思路演进:好位置的传递

定义"**好位置**"为:从该位置出发能到达终点的位置。

1. 终点 `n-1` 自己就是好位置
2. 从右往左扫,如果 `nums[i] >= goodPos - i`(从 i 能跳到当前最靠左的好位置),那么 i 也是好位置
3. 把 `goodPos` 更新为 i,继续往左
4. 最终如果 `goodPos == 0`,说明起点是好位置 → 能到达终点

### 走一遍 `[2,3,1,1,4]`

初始 `goodPos = 4`

| i | nums[i] | goodPos - i | nums[i] >= 距离? | 更新后 goodPos |
|---|---------|-------------|------------------|----------------|
| 3 | 1 | 1 | ✅ | 3 |
| 2 | 1 | 1 | ✅ | 2 |
| 1 | 3 | 1 | ✅ | 1 |
| 0 | 2 | 1 | ✅ | 0 |

最终 `goodPos == 0`,返回 `true` ✅

### 反例 `[3,2,1,0,4]`

初始 `goodPos = 4`

| i | nums[i] | goodPos - i | nums[i] >= 距离? | 更新后 goodPos |
|---|---------|-------------|------------------|----------------|
| 3 | 0 | 1 | ❌ | 4 |
| 2 | 1 | 2 | ❌ | 4 |
| 1 | 2 | 3 | ❌ | 4 |
| 0 | 3 | 4 | ❌ | 4 |

最终 `goodPos == 4 != 0`,返回 `false` ✅(卡在 nums[3]=0)

## 代码实现:逆向贪心

```java
class Solution {
    public boolean canJump(int[] nums) {
        int n = nums.length;
        int goodPos = n - 1;  // 终点是初始的好位置

        // 从倒数第二个位置开始往前扫
        for (int i = n - 2; i >= 0; i--) {
            // 如果从 i 能跳到 goodPos,那 i 也是好位置
            if (i + nums[i] >= goodPos) {
                goodPos = i;
            }
        }

        return goodPos == 0;
    }
}
```

**复杂度**:时间 O(n),空间 O(1)

## 替代解法:正向贪心

维护"当前能到达的最远距离" `maxReach`:

```java
class Solution {
    public boolean canJump(int[] nums) {
        int maxReach = 0;
        for (int i = 0; i < nums.length; i++) {
            if (i > maxReach) return false;  // 到不了 i
            maxReach = Math.max(maxReach, i + nums[i]);
        }
        return true;
    }
}
```

**复杂度**:时间 O(n),空间 O(1)

## 易错点

1. **变量声明语法**:`int n = nums.length;` 不是 `int nums.length = n;`
2. **判断条件不是 `nums[i] >= i`**:应该是 `nums[i] >= goodPos - i`,因为 `i` 不是距离,`goodPos - i` 才是距离
3. **不需要严格相等**:`nums[i] >= 距离` 即可,跳过头也算到达

## Jump I 总结

1. **核心思想**:好位置可以从终点向左传递
2. **判断条件**:`i + nums[i] >= goodPos` → i 也是好位置
3. **终止条件**:`goodPos == 0` 返回 true,否则 false
4. **两种实现**:逆向(从右往左推 goodPos)或正向(维护 maxReach)
5. **复杂度**:都是 O(n) 时间,O(1) 空间

# 45. Jump Game II

## 题目描述

给定一个非负整数数组 `nums`,初始位于第 0 个位置。每个 `nums[i]` 是从 i 出发能跳的最大长度。

**返回**:到达最后一个位置所需的**最少跳跃次数**。题目保证一定能到达终点。

**示例**:
```
Input:  nums = [2,3,1,1,4]
Output: 2   // 0 -> 1 -> 4
```

## 关键问题:能否复用 Jump I 的逆向思路?

**答案**:理论上可以,但会退化成 **O(n²) 的 DP**:

```java
// dp[i] = 从 i 到终点的最少步数
dp[n-1] = 0;
for (int i = n - 2; i >= 0; i--) {
    dp[i] = INF;
    for (int j = 1; j <= nums[i] && i + j < n; j++) {
        dp[i] = Math.min(dp[i], dp[i + j] + 1);
    }
}
return dp[0];
```

**Jump II 的最优解是正向贪心(BFS 分层)**,因为求"最少步数"需要"分层"的概念。

## 正向贪心:BFS 分层思想

把每一步能到达的范围看成 BFS 的一层:

```
nums = [2, 3, 1, 1, 4]

第 0 层 (0 步可达): {0}                    起点
                     ↓ 跳!
第 1 层 (1 步可达): {1, 2}                 从 0 出发最远到 2
                     ↓ 跳!
第 2 层 (2 步可达): {3, 4, ...}            从 [1,2] 中最远跳到 4 ← 终点!

答案 = 2 步
```

## 三个关键变量

| 变量 | 含义 | 类比 |
|------|------|------|
| `currentEnd` | **当前步**能到达的最远位置(已确定的层边界) | 防线指挥官的位置 |
| `farthest` | **下一步**能到达的最远位置(侦察累积) | 侦察兵看到的远方 |
| `jumps` | 已跳跃次数 | 行军令消耗数 |

### 核心逻辑:三种情况

```
当 i < currentEnd:
    ★ 还在当前层内,只更新 farthest 进行侦察
    ★ 不动 currentEnd 和 jumps

当 i == currentEnd:
    ★ 当前层走完了,必须跳一步
    ★ jumps++ (新的一步开始)
    ★ currentEnd = farthest (把侦察结果当作新一层的右边界)
    ★ farthest 不重置(单调不减,继续累积)
```

## 代码实现:正向贪心

```java
class Solution {
    public int jump(int[] nums) {
        int n = nums.length;
        int jumps = 0;
        int currentEnd = 0;   // 当前步数能到达的最右边界
        int farthest = 0;     // 下一步能到达的最远位置

        // 注意:i 只到 n-2,因为到达 n-1 就结束了
        for (int i = 0; i < n - 1; i++) {
            // 侦察:更新下一步能到达的最远位置
            farthest = Math.max(farthest, i + nums[i]);

            // 撞到当前层右边界,必须跳一步
            if (i == currentEnd) {
                jumps++;
                currentEnd = farthest;
            }
        }

        return jumps;
    }
}
```

**复杂度**:时间 O(n),空间 O(1)

## 走一遍 `[2,3,1,1,4]`

初始:`jumps=0, currentEnd=0, farthest=0`

| i | nums[i] | i+nums[i] | farthest | i==currentEnd? | jumps | currentEnd | 动作 |
|---|---------|-----------|----------|----------------|-------|------------|------|
| 0 | 2 | 2 | 2 | ✅ (0==0) | 1 | 2 | **跳!确定 1 步范围 [1,2]** |
| 1 | 3 | 4 | 4 | ❌ | 1 | 2 | 侦察:下一层最远到 4 |
| 2 | 1 | 3 | 4 | ✅ (2==2) | 2 | 4 | **跳!确定 2 步范围 [3,4]** |
| 3 | 1 | 4 | 4 | ❌ | 2 | 4 | (循环结束) |

返回 `jumps = 2` ✅

## 关键问题深入剖析

### Q1: 为什么 `i + nums[i]` 不需要限定 `< n`?

因为 `i + nums[i]` 是**纯数值**,只用于赋值给 `farthest`,**不会用作数组下标**。

| 变量 | 用途 | 是否需要边界检查 |
|---|---|---|
| `i` | 数组下标,访问 `nums[i]` | ✅ 必须 (`i < n`) |
| `i + nums[i]` | 数值,赋给 `farthest` | ❌ 不需要 |
| `farthest` | 数值,赋给 `currentEnd` | ❌ 不需要 |
| `currentEnd` | 数值,与 `i` 比较 | ❌ 不需要 |

**举例 `nums = [1, 22, 11, 1000, 3, 4]`**:`farthest` 一度飙到 1003(远超 n=6),但因为它只是数值,且循环条件 `i < n-1` 保证了 i 不会越界,所以**完全无害**。

**核心原则**:只有"用作数组下标"的变量才需要边界检查,纯数值计算/比较不需要。

### Q2: 为什么循环条件是 `i < n - 1` 而不是 `i < n`?

因为站在终点(`i = n-1`)时**不需要再跳了**。如果写成 `i < n`,最后一步会被多算一次。

**反例**:`nums = [1]`,正确答案是 0(已在终点)。如果循环到 `i = 0`,会触发 `i == currentEnd`,`jumps` 错误地变成 1。

### Q3: 为什么 `i == currentEnd` 时才 `jumps++`?

`currentEnd` 是"当前步数能到的最远位置",当 `i` 撞到这个边界时,意味着:
- **当前这一步用完了**,如果还想前进,必须跳一步
- 此时 `farthest` 已经积累了"当前层所有位置出发能到的最远",可以作为新边界

**类比"防线推进"**:
- 侦察兵(`farthest`)不停向前张望,记录"我们能打到的最远地方"
- 指挥官(`currentEnd`)在大本营坐镇,**只在收到"当前防线全部探查完毕"的报告时**,才下令推进防线
- **必须等当前防线内的所有位置都探查完,才能推进,且只算一次行军**

如果每发现更远的位置就 `jumps++`,会**高估步数**——因为同一层内的多次"侦察更远",其实是在用**同一次跳跃**的预算。

### Q4: `farthest` 和 `currentEnd` 的关系?它们都是 max 比较的结果吗?

**这是最容易混淆的点!**两者维护的是**两层不同的信息**:

| 变量 | 含义 | 更新频率 | 更新公式 |
|------|------|----------|----------|
| `farthest` | **下一步**能到的最远位置 | **每个 i 都更新** | `max(farthest, i + nums[i])` |
| `currentEnd` | **当前步**能到的最远位置 | **只在 i == currentEnd 时更新** | `currentEnd = farthest` |

**关键差异**:
1. `farthest` 是 `max(farthest, ...)`,**不是** `max(currentEnd, ...)`
2. 是 `i + nums[i]`(站在 i 出发能跳多远),**不是** `currentEnd + nums[i]`
3. `currentEnd` 不是每次都更新,**只在到达层边界时一次性更新**

**反例分析**:如果写成 `currentEnd = max(currentEnd, i + nums[i])`,在 `i=1` 时 `currentEnd` 会立刻变成 4,相当于"没用 1 步就到了 4",`jumps` 会算少,结果错误。

### Q5: `farthest` 跨层时为什么不重置?

因为:
1. **`farthest` 是单调不减的**(`Math.max` 只会让它变大)
2. 上一层尾端能跳到的位置,跟下一层开头能跳到的位置,是**累积关系**而非替换关系
3. 重置反而会丢失信息

## 易错点对比

| 错误写法 | 后果 |
|---|---|
| `for (int i = 0; i < n; i++)` | 终点会被多算一步 |
| 每个 i 都 `jumps++` | 等于步数=数组长度,错误 |
| `farthest` 更新就 `jumps++` | 把"侦察更远"和"实际跳一步"混淆 |
| `currentEnd = max(currentEnd, i + nums[i])` | `jumps` 算少,结果错误 |
| 跨层时重置 `farthest = 0` | 丢失累积信息 |

## Jump II 总结

1. **核心思想**:BFS 分层贪心,每一层是"用 k 步能到的所有位置"
2. **三个变量**:`currentEnd`(当前层右边界)、`farthest`(下一层最远位置侦察值)、`jumps`(步数)
3. **核心节奏**:`farthest` 每步侦察,`currentEnd` 只在 `i == currentEnd` 时跃迁
4. **触发条件**:`i == currentEnd` → `jumps++` 且 `currentEnd = farthest`
5. **循环边界**:`i < n - 1`(终点不需要再跳)
6. **复杂度**:时间 O(n),空间 O(1)

# Jump I vs Jump II 对比

| 维度 | Jump I (能否到达) | Jump II (最少步数) |
|---|---|---|
| **目标** | true / false | 最小整数 |
| **关键变量** | `goodPos` 或 `maxReach` | `currentEnd` + `farthest` + `jumps` |
| **为什么 Jump II 多变量** | 不需要分层 | 需要区分"当前层"和"下一层" |
| **核心思想** | 好位置传递 / 最远可达性 | BFS 分层贪心 |
| **逆向 DP 可行性** | O(n) 最优解 | O(n²),非最优 |
| **正向贪心** | 维护 maxReach | 维护 currentEnd + farthest |

## 思维模型迁移

这两道题的思路可以推广到一类**"区间跳跃 / 区间覆盖"问题**:

- **LeetCode 1024 视频拼接**:用最少视频段覆盖区间
- **LeetCode 1326 浇花问题**:用最少水龙头覆盖花园
- **LeetCode 1306 跳跃游戏 III**:能否到达任意 `nums[i] == 0` 的位置

**通用模式**:
1. 把问题建模为"用最少的区间覆盖目标"
2. 维护"当前覆盖的右边界" + "下一段能延伸到的最远"
3. 当 `i` 撞到右边界时,选择能延伸最远的一段,边界推进 + 计数

# 最终知识点回顾

1. **Jump I**:`i + nums[i] >= goodPos` 时 `i` 是好位置;`goodPos` 从 `n-1` 向左收敛到 0 即为 true
2. **Jump II**:`farthest` 不停侦察,`currentEnd` 只在 `i == currentEnd` 时跃迁,`jumps` 同步 +1
3. **`farthest` 和 `currentEnd` 是两层信息**,更新节奏不同,不能混为一谈
4. **数值 vs 下标**:只有用作数组下标的变量需要边界检查
5. **循环条件 `i < n - 1`**:站在终点时不需要再跳
6. **思维模型**:BFS 分层贪心是"区间跳跃"类问题的通用范式
